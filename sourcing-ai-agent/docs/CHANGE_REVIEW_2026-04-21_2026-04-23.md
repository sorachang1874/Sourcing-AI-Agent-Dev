# Change Review: 2026-04-21 To 2026-04-23

> Status: Current first-party doc. Treat this file as active guidance, but keep it aligned with `docs/INDEX.md` and `PROGRESS.md` when runtime contracts change.


This note re-judges the main changes from the 2026-04-21 to 2026-04-23 stabilization pass against the current codebase.

The point is not to preserve every intermediate fix. The point is to keep the changes that moved the system toward a single contract, and keep unwinding the ones that were still local patches or compatibility tails.

## Changes That Were Correct

- Shared contract extraction instead of caller-local gates.
  - Examples:
    - baseline candidate selection / authoritative promotion sharing logic
    - `provider_execution_policy.py` for core provider-lane semantics
    - execution semantics shared across planner / runtime / explain
  - Reason:
    - these removed the recurring planner/runtime/profile drift that kept reappearing in manual testing.

- PG-first control plane and physical former/current split.
  - Reason:
    - this was a real architectural cutover, not a cosmetic wrapper.
    - it removed a major source of silent divergence between live writes, migrations, and reads.

- Summary-first API payloads.
  - Reason:
    - `/api/jobs/{job_id}` and `/results` were genuinely too heavy for polling and board hydration.
    - shrinking default payloads fixed a structural bottleneck instead of hiding it in the frontend.

- Deferred post-stage materialization after authoritative results are already ready.
  - Reason:
    - moving snapshot reconcile/materialization behind the foreground result contract was the right fix for “Final Results displayed but board still blocked”.

- Replay/scripted cache pollution cleanup.
  - Reason:
    - treating replay as cache-only and stopping `_offline` payloads from entering shared live caches fixed a real environment isolation bug.

- Semantic-brief driven planning refactor.
  - Reason:
    - replacing duplicated `_infer_*` branches with a clearer `intent_view -> semantic_brief -> execution semantics -> task projection` chain is the right long-term direction.

## Changes That Were Directionally Right But Still Incomplete

- `allow_high_cost_sources` demotion.
  - Correct direction:
    - core roster/profile-search lanes should not depend on this flag.
  - Remaining tail:
    - compatibility plumbing still exists in review/preferences payloads.
    - it should stay compatibility-only and must not regain core planning power.

- Snapshot reuse rule cleanup.
  - Correct direction:
    - “many snapshots” should not be the decision rule.
    - “authoritative coverage proof” should be the decision rule.
  - Remaining tail:
    - some paths still rely on intermediate proxies such as selection mode rather than a first-class promoted aggregate contract.

- Stage timing and board readiness cleanup.
  - Correct direction:
    - hidden finalize work must not block the foreground result contract.
  - Remaining tail:
    - candidate hydration and staged board rendering still need stricter timing/reporting assertions.

## Changes That Were Not Good Enough And Need Continued Rollback/Cleanup

- Generic flags overriding specific acquisition semantics.
  - Bad pattern:
    - `allow_high_cost_sources` or generic cost gates deciding whether scoped/full/former Harvest lanes run.
  - Better reference method:
    - specialized contract wins; generic flags only affect optional augmentations.

- Core lane semantics described as web-search-first.
  - Bad pattern:
    - planner/review/explain metadata saying `web_search_seed_queries` is the primary roster source for scoped/former core lanes.
  - Better reference method:
    - user-visible plan metadata should describe the actual authoritative provider lane first, not historical helper sources.

- Hidden dual-track fallbacks.
  - Bad pattern:
    - PG-only path that still silently falls back to SQLite-shaped behavior or shared runtime artifacts.
  - Better reference method:
    - if compatibility exists, document it explicitly and prevent it from becoming live truth.

- Environment handling by repeated ad hoc rediscovery.
  - Bad pattern:
    - after context compaction, re-detecting shell/workdir/venv from scratch and wasting time on the same probes.
  - Better reference method:
    - resume rule must be explicit:
      - read latest interaction record
      - read `PROGRESS.md`
      - verify workdir
      - run tests with repo venv

## Reference Methods To Reuse When “Changing Back”

- When a fix needs to touch planner, runtime, explain, and recovery:
  - extract a shared helper or contract first.

- When a fix only makes one visible surface correct:
  - assume it is incomplete until plan/progress/results agree on the same semantics.

- When a compatibility path remains:
  - keep it read-compatible only, document it, and add a removal TODO.

- When a runtime/test/live environment differs:
  - isolate by runtime namespace, not only by provider mode flags.

- When performance is bad:
  - first separate:
    - provider wait
    - materialization
    - result assembly
    - frontend hydration
  - then optimize the dominant stage instead of shrinking payloads or masking timers.

## What This Review Implies For The Next Cleanup Pass

- Keep moving core acquisition semantics toward a single contract.
- Continue removing misleading web-search wording from core lane metadata.
- Continue replacing compatibility flags with explicit targeted contracts.
- Treat frontend-visible timing/count regressions as first-class regression tests, not manual QA only.
- Keep environment bootstrap and runtime isolation as architecture work, not operator folklore.
